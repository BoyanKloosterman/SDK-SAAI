1. UML Klassendiagram
Wat is een klasse?
Een klasse is een blauwdruk voor objecten. In UML teken je hem als een rechthoek met drie vakjes.
- (min) = private → alleen de klasse zelf mag erbij.
+ (plus) = public → iedereen mag erbij.
# (hekje) = protected → alleen de klasse en subklassen mogen erbij.
De 6 UML-relaties — het grote overzicht
Welk lijntje gebruik je voor welke relatie? Dit is het meest gevraagde op de toets.
🔑 Ezelsbruggetje compositie vs aggregatie: Stel jezelf de vraag — "Als de parent verdwijnt, verdwijnt de child dan ook?" Ja → compositie ◆. Nee → aggregatie ◇.
Hoe kies je de juiste relatie bij een casus?
Stap 1: Lees de zin. "Een Order heeft OrderItems." → Order en OrderItem.
Stap 2: Kan een OrderItem bestaan zonder Order? Nee → compositie ◆.
Stap 3: Kan een Speler bestaan zonder Team? Ja → aggregatie ◇.
Stap 4: Verwijst klasse A naar klasse B als attribuut, maar ze leven onafhankelijk? → associatie →.
2. Design principes & Design Smells
SRP — Single Responsibility Principle
Elke klasse heeft maar één taak en dus maar één reden om te veranderen.
Voorbeeld fout: UserManager maakt gebruikers aan, stuurt mails én slaat op in de database. Dat zijn drie taken → drie klassen!

Voorbeeld goed: UserService (aanmaken), EmailService (mailen), UserRepository (opslaan).
S in SOLID
OCP — Open/Closed Principle
Een klasse moet open zijn voor uitbreiding, maar gesloten voor aanpassing. Dus: nieuwe functionaliteit toevoegen zonder bestaande code te veranderen.
Voorbeeld fout:
if shape == "Circle" → drawCircle()
if shape == "Square" → drawSquare()
Als je een nieuw shape wilt, moet je deze methode aanpassen. Dat is slecht!

Voorbeeld goed: Maak een interface IShape met een draw() methode. Elke shape (Circle, Square) implementeert die zelf. Nieuw shape? Gewoon een nieuwe klasse toevoegen, niets aanpassen.
O in SOLID
Loose Coupling — los van elkaar
Een klasse is niet afhankelijk van een concrete klasse, maar van een interface. Zo kun je makkelijk wisselen van implementatie.
Waarom is dit handig? Stel je wil van MySQL naar XML overstappen. Bij tight coupling moet je OrderService aanpassen. Bij loose coupling hoef je alleen een nieuwe implementatie te maken — OrderService blijft onveranderd.
Design for Change — bouw voor de toekomst
Software verandert altijd. Als je niet nadenkt over uitbreidbaarheid, wordt aanpassen steeds moeilijker en sterft je software langzaam.
Simpel gezegd: Schrijf code alsof je over een jaar een nieuwe collega hebt die er iets aan moet aanpassen. Begrijpt die het nog? Kan die het makkelijk uitbreiden?
OnderhoudUitbreidbaarheid

3. Applicatiearchitectuur & Domeinmodel
Design vs Architectuur — wat is het verschil?
🔧 Design
Hoe bouw je één klasse of methode? Dichter bij de code. Details zoals welke klassen er zijn en hoe ze samenwerken.
🏗️ Architectuur
Hoe is het hele systeem opgebouwd? Groter plaatje. Opdeling in subsystemen, lagen en modules.
Makkelijk onthouden: Design = je denkt aan één kamer inrichten. Architectuur = je denkt aan hoe het hele huis is gebouwd.
Drie lagen van een applicatie
Elke applicatie heeft drie verantwoordelijkheden. Die stop je in aparte lagen.
Voorbeeld: Een webshop. Presentation = de website die je ziet. Domain = de regels: "je kunt niet meer bestellen dan er op voorraad is." Data source = de database met producten en bestellingen.
Domeinmodel — het hart van de app
Het domeinmodel beschrijft de essentiële objecten en regels van de wereld waarvoor je bouwt. Je modelleert het als een UML klassendiagram.
Voorbeeld: Bouw je een bibliotheekapp? Dan is het domeinmodel: Boek, Lid, Exemplaar, Uitlening — en de regels daartussen (een lid mag max. 5 boeken lenen, etc.).

4. Systeemarchitectuur & Analyse
Applicatie- vs systeemarchitectuur
📱 Applicatiearchitectuur
Hoe is één applicatie van binnen opgebouwd? Lagen, modules, klassen. Voorbeeld: een webshop-app met presentation, domain en data-laag.
🌐 Systeemarchitectuur
Hoe werken meerdere systemen samen? Voorbeeld: een webshop-systeem met een frontend, backend, betaalsysteem en database die via APIs communiceren.
Architectuuranalyse — waarom moet je dit doen?
Voordat je iets bouwt of aanpast in een bestaand systeem, moet je het systeem begrijpen.
⚠️ Als je dit overslaat: je schrijft code die al bestaat, je breekt dingen die al werkten, en je maakt keuzes die eerder al zijn afgewezen.
In de praktijk: Op school bouw je een app vanaf nul. Op stage of bij een bedrijf stap je bijna altijd in een bestaand systeem dat al jaren loopt. Je moet dit eerst begrijpen voordat je iets aanpast.

5. Messaging, Events & Communicatie
Synchroon vs asynchroon — het grote verschil
📞 Synchroon
A belt B op en wacht totdat B opneemt en antwoordt. Als B niet bereikbaar is, mislukt de hele operatie.

Voorbeeld: REST API call — je wacht op het antwoord voordat je verder gaat.
📬 Asynchroon
A stopt een berichtje in de brievenbus en gaat verder. B leest het wanneer die er klaar voor is.

Voorbeeld: Message in een queue sturen — je hoeft niet te wachten op een antwoord.
Wanneer kies je asynchroon? Als systemen niet altijd tegelijk online zijn, als een taak lang duurt, of als meerdere systemen moeten reageren op één gebeurtenis.
Message vs Event — wat is het verschil?
✉️ Message
Een gerichte opdracht van A naar B. A verwacht dat B iets doet.

"Verwerk deze betaling."
📢 Event
Een aankondiging dat er iets is gebeurd. Niemand hoeft er per se op te reageren.

"De betaling is verwerkt."
Ezelsbruggetje: Message = toekomst (er moet iets gaan gebeuren). Event = verleden (er is iets gebeurd).
Queue — de wachtrij
Een queue is een wachtrij voor berichten. Berichten gaan er in (FIFO = eerst in, eerst eruit) en consumers lezen ze op hun eigen tempo.
✅ Voordeel: Als de Consumer even offline is, gaan de berichten niet verloren. Ze wachten gewoon in de queue totdat hij er klaar voor is.
Temporal & Behavioral Coupling
⏰ Temporal coupling
Probleem: A en B moeten tegelijk online zijn om te communiceren.

Oplossing: Queue — A stuurt het bericht, de queue bewaart het, B leest het later.
🔗 Behavioral coupling
Probleem: A weet precies wat B doet en verwacht een specifieke reactie.

Oplossing: Events — A roept alleen "dit is gebeurd", B beslist zelf wat te doen.

6. Applicatie-integratie & Architectuurstijlen
Waarom is applicatie-integratie een probleem?
Grote organisaties gebruiken tientallen systemen: HR-systeem, CRM, webshop, boekhoudpakket. Die zijn allemaal los gebouwd en praten slecht met elkaar.
⚠️ Typische problemen: data-silo's (data zit opgesloten in één systeem), dubbele data (klantgegevens staan in drie systemen), handmatige koppelingen via Excelbestanden → fouten, inefficiëntie.
Communicatiemethoden — hoe praten systemen met elkaar?
MethodeHoe werkt het?Wanneer gebruik je het?
RESTHTTP-verzoeken (GET/POST/PUT/DELETE) met JSON-antwoord. Simpel en breed ondersteund.Moderne APIs, de meest gebruikte methode
SOAPStrikte berichten in XML-formaat. Veel regels en structuur.Oudere bedrijfssystemen die dit nog gebruiken
Messaging (queue)Berichten via een broker/queue. Asynchroon.Als systemen los van elkaar moeten werken of de belasting hoog is
Point-to-point koppeling — directe verbinding
Systemen praten rechtstreeks met elkaar. Snel opgezet, maar bij veel systemen wordt het een spaghetti.
✅ Voordelen
Snel te bouwen, simpel te begrijpen bij 2 systemen
❌ Nadelen
Bij 5 systemen heb je al 10+ verbindingen. Tight coupling, niet schaalbaar, moeilijk te onderhouden
💡 De oplossing: stop met denken in systemen die met elkaar praten, en denk in services die functionaliteit aanbieden.
SOA — Service Oriented Architecture
Alle functionaliteit wordt aangeboden als losse herbruikbare services. Die services communiceren via een centrale ESB (Enterprise Service Bus).
Voordeel: Services zijn herbruikbaar. De Payments-service kan door zowel de webshop als de mobile app worden gebruikt.
Nadeel: De ESB wordt een centrale bottleneck — als die uitvalt, valt alles uit. En het is complex om te beheren.

7. Snelle begrippen
Abstracte klasse vs Interface — het cruciale verschil
📦 Abstracte klasse
Kan gedeelde code bevatten én abstracte methoden die subklassen verplicht moeten implementeren.

Je kunt maar van één abstracte klasse erven.

abstract class Shape { abstract draw(); }
📋 Interface
Bevat alleen handtekeningen — geen code, alleen de afspraken. Een klasse kan meerdere interfaces tegelijk implementeren.

interface IFeedable { feed(); }
🔑 Onthoud: Je kunt maar van één abstracte klasse erven, maar je kunt meerdere interfaces tegelijk implementeren. Dit is het meest gevraagde verschil!

8. Loose Coupling & Factory Pattern
Wat is coupling?
Coupling = hoe sterk twee klassen aan elkaar vastzit. Loose coupling = zo min mogelijk afhankelijk van elkaar. Tight/strong coupling = heel erg van elkaar afhankelijk, moeilijk te wijzigen.
Probleem tight coupling:
MemberDAO memberDAO = new MySqlMemberDAO();
Je noemt hier de concrete klasse MySqlMemberDAO. Als je wisselt naar een andere database, moet je alle code aanpassen. Slecht!

Oplossing — program to an interface:
MemberDAO memberDAO; // gebruik het interface-type!
Je weet alleen dat het een MemberDAO is, niet welke concrete versie.
🔑 Onthoud: new ConcreteKlasse() = altijd tight coupling. Dat kun je niet vermijden — maar je kunt het beperken tot zo min mogelijk plekken in je code.
Dependency Injection (DI)
In plaats van zelf een object aanmaken met new, geef je het object mee via de constructor. Zo hoeft een klasse niet te weten welke concrete implementatie hij gebruikt.
Zonder DI (tight coupling):
MemberDAO dao = new MySqlMemberDAO(); // hardcoded!

Met DI (loose coupling):
public MemberAdminManager(MemberDAO memberDAO) {
  this.memberDAO = memberDAO;
}

De klasse die MemberAdminManager aanmaakt, beslist welke concrete DAO het wordt. MemberAdminManager zelf weet het niet!
Het Factory Pattern — waarom?
Elke keer dat je new schrijft, heb je tight coupling. Je kunt dit niet vermijden, maar je kunt het beperken tot één plek in je code: de factory.
Doel van een factory:
• Maakt objecten aan op één centrale plek
• Schermt af om welke concrete klasse het gaat
• Retourneert het object als interface (zodat de rest van je code loose coupled blijft)
• Een factory zelf is dus wél tight coupled — dat is de bedoeling!
🏭 Simple Factory
Eén klasse die op basis van een parameter bepaalt welk concreet object aangemaakt wordt.

switch(source) {
case CSV → new CSVMemberDAO();
case SQL → new SQLMemberDAO();
}

Nadeel: als je een nieuwe databron toevoegt, moet je de factory aanpassen → schendt OCP!
🏭🏭 Abstract Factory
Er is een interface voor de factory zelf. Elke concrete factory maakt alle DAO's voor één bepaalde databron aan.

DAOFactory → SQLServerDAOFactory
DAOFactory → TestDataDAOFactory

Voordeel: je kunt makkelijk een nieuwe databron toevoegen (nieuwe factory klasse) zonder bestaande code te wijzigen → voldoet aan OCP!
Factory Instantiatie — hoe krijg je een factory?
De manager heeft een factory nodig, maar wie maakt de factory aan? Dat doet de main klasse!
Stap voor stap:
1. main maakt een concrete factory aan: new SQLServerDAOFactory()
2. main geeft de factory via de constructor aan de manager mee (dependency injection!)
3. De manager heeft een attribuut van type DAOFactory (de interface) → blijft loose coupled
4. De manager vraagt de factory om DAO's: factory.createMemberDAO()
⚠️ Beperking: Met één factory heb je maar één actieve databron voor je hele app. Alle DAO's zijn dan MySQL óf allemaal TestData. Je kunt niet mixen.

9. Doorontwikkeling, SRP & Refactoring
SRP — Single Responsibility (verdieping)
Een klasse heeft maar één verantwoordelijkheid. Maar hoe weet je wat die verantwoordelijkheid is? Vraag jezelf af: "Waarom zou ik deze klasse moeten aanpassen?"
Voorbeeld BSN-validatie:
Stel je voegt een BSN (BurgerServiceNummer) toe aan een Member. Je zet de validatiemethode in de Member-klasse. Maar klopt dat?

Nee! De Member-klasse gaat over een lid van de bibliotheek. Validatielogica voor BSN is een aparte verantwoordelijkheid. Maak een aparte klasse: BurgerServiceNummer.

Voordeel: Als de BSN-regels veranderen, pas je alleen de BurgerServiceNummer klasse aan — niet de Member-klasse.
❌ Fout: alles in Member
class Member {
  String bsn;
  boolean validateBSN() { ... }
}
Member heeft nu 2 redenen om te veranderen: lidmaatschapslogica én BSN-validatielogica.
✅ Goed: aparte klasse
class BurgerServiceNummer {
  String nummer;
  boolean isGeldig() { ... }
}

class Member {
  BurgerServiceNummer bsn;
}
Internationalisering — SSN (Social Security Number)
Stel je app gaat internationaal. Een Nederlands BSN is anders dan een Amerikaans SSN. Hoe los je dit op zonder je hele code om te gooien?
Oplossing: Maak een abstracte klasse of interface SSN en maak concrete subklassen per land:
• BurgerServiceNummer (Nederland)
• SocialSecurityNumber (VS)

De Member klasse krijgt een attribuut van type SSN (de interface) + een country attribuut.

Om te bepalen welke concrete SSN aangemaakt moet worden, gebruik je een Simple Factory:
SSN ssn = SsnFactory.create(country);
💡 Wie gebruikt de SsnFactory? Zowel de GUI (bij het aanmaken van een Member) als de DAO (bij het ophalen uit de database). Zo voorkom je code-duplicatie!
Refactoring — wat is het?
Refactoring = de interne structuur van code verbeteren zonder dat de functionaliteit verandert. Je maakt de code cleaner, maar het gedrag blijft hetzelfde.
Voorbeelden van refactoring:
• Extract Interface: Maak een interface van een klasse zodat je loose coupling kunt toepassen
• Extract Method: Haal een stuk code uit een lange methode en maak er een aparte methode van
• Extract Class: Een klasse doet te veel → splits op in twee klassen (SRP!)
• Rename: Geef variabelen en methoden duidelijkere namen

Hoe? IDE-tools (IntelliJ, Eclipse) helpen je refactoren zonder fouten te maken. Bijv. Extract Interface doet alles automatisch.
🎯 Doel van refactoring
• Leesbaarheid vergroten
• Technical debt verminderen
• SRP beter naleven
• Loose coupling bereiken
• Onderhoud makkelijker maken
⚠️ Let op!
Refactoring verandert nooit het gedrag van de software! Je test daarna of alles nog werkt. Altijd refactoren vóór je nieuwe features toevoegt — dan wordt de codebase niet steeds rommeliger.
Cohesion — verwant aan SRP
Cohesion (samenhang) = in hoeverre horen de verantwoordelijkheden van een klasse bij elkaar? Hoge cohesie = goed. Lage cohesie = de klasse doet te veel ongerelateerde dingen.
Lage cohesie (slecht): Een klasse die BSN valideert, e-mails verstuurt én in de database opslaat.

Hoge cohesie (goed): Elke klasse doet één ding heel goed — zoals BurgerServiceNummer die alleen over BSN-nummers gaat.
🔑 Verband SRP ↔ Cohesie: Als je SRP goed toepast, krijg je automatisch hoge cohesie. Het zijn twee kanten van dezelfde munt.

10. Nygard ADR — het exacte format
Waarom het Nygard-format?
Op de toets moet je zelf een ADR schrijven in Nygard-format. Niet uitleggen wat het is — maar het daadwerkelijk invullen. Hieronder het exacte format dat je moet kennen.
⚠️ Let op: Er zijn meerdere ADR-formats (MADR, Nygard). Op de toets wordt specifiek het Nygard-format gevraagd. Verschil: Nygard heeft géén "Decision Drivers" of "Considered Options" als aparte sectie — dat zit verpakt in Context.
Het Nygard-template — letterlijk
Elke ADR heeft precies 5 secties. Niets meer, niets minder.

Nygard ADR — ingevuld voorbeeld
Zo ziet een complete Nygard-ADR eruit bij een messaging-keuze:
# Gebruik van RabbitMQ als message broker

## Status
Accepted

## Context
De nieuwe betalingsmodule moet orders verwerken zonder afhankelijk te zijn van de beschikbaarheid van het legacy-systeem. Bij direct koppelen (REST) valt alles uit als het legacy-systeem offline is (temporal coupling). We hebben een oplossing nodig die berichten bewaart totdat ze verwerkt kunnen worden.

## Decision
We gebruiken RabbitMQ als message broker. De betalingsmodule plaatst betalingsopdrachten als messages in een queue. Het legacy-systeem leest deze messages asynchroon op.

## Consequences
+ Temporal coupling opgelost: systemen hoeven niet tegelijk online te zijn.
+ Berichten gaan niet verloren bij uitval (durable queue).
- Extra infrastructuur: RabbitMQ-server moet beheerd worden.
- Complexer debuggen: je kunt niet direct zien of een bericht verwerkt is.

11. Abstract Factory — stap 1 vs stap 2
Stap 1 vs Stap 2 — wat is het verschil?
In de workshops worden twee stappen van het factory-pattern behandeld. Dit is precies wat de toets vraagt.
🏭 Stap 1 — Simple Factory
Eén concrete factory-klasse met een switch/if die het juiste object aanmaakt.

Probleem: als je een nieuwe soort toevoegt, moet je de factory aanpassen → schendt OCP!

class DAOFactory {
 createDAO(type) {
  if SQL → new SqlDAO()
  if XML → new XmlDAO()
 }}
🏭🏭 Stap 2 — Abstract Factory
Een interface voor de factory zelf. Elke concrete factory-klasse maakt objecten voor één "familie".

Voordeel: Nieuwe familie toevoegen = nieuwe factory-klasse. Bestaande code hoeft niet aan te worden geraakt → OCP ✅

interface DAOFactory
SqlDAOFactory implements DAOFactory
XmlDAOFactory implements DAOFactory
Stap 2 diagram — de volledige structuur
Dit is het patroon dat je moet kunnen tekenen en uitleggen. Let op de twee "kanten": de factory-kant en de product-kant.
Bibliotheek-voorbeeld:
• DAOFactory = factory-interface
• SQLServerDAOFactory = maakt SQL-varianten: SQLMemberDAO, SQLBookDAO
• TestDataDAOFactory = maakt test-varianten: TestMemberDAO, TestBookDAO
• De Manager-klasse weet alleen van DAOFactory → volledig loose coupled
Hoe voeg je een nieuw monster/product toe? (toets-vraag!)
Dit is precies wat vraag 1 van de oefentoets vraagt. Leer dit stappenplan.
Stel: je voegt een nieuw MonsterType (Gorilla) en een nieuwe reeks (SilverLine) toe.

Nieuwe monster-soort (Gorilla):
1. Maak interface of abstracte klasse Gorilla (naast Spider, Duck, Tiger)
2. Voeg createGorilla() : Gorilla toe aan de MonsterFactory-interface
3. Maak voor elke bestaande reeks een concrete klasse: FrostyGorilla, FieryGorilla, GoldGorilla
4. Voeg createGorilla() toe aan elke bestaande concrete factory

Nieuwe reeks (SilverLine):
1. Maak SilverFactory die MonsterFactory implementeert
2. Implementeer alle create-methoden: createSpider() → new SilverSpider() etc.
3. Maak de bijbehorende concrete klassen: SilverSpider, SilverDuck, SilverTiger
💡 Verschil: Nieuwe soort toevoegen = aanpassen van factory-interface + bestaande factories (schendt eigenlijk OCP). Nieuwe reeks toevoegen = alleen nieuwe klassen → volledig OCP-proof! Dit is waarom Abstract Factory zo krachtig is voor het toevoegen van nieuwe "families".
12. Delivery Guarantees — hoe zeker is je bericht?
De drie delivery guarantees
Wanneer je een bericht stuurt via een queue, is er geen garantie dat het altijd precies één keer aankomt. Er zijn drie niveaus van zekerheid.
GuaranteeWat betekent het?RisicoWanneer gebruik je het?
At most once
hooguit één keer		Het bericht wordt maximaal één keer afgeleverd. Als het mislukt, wordt het niet opnieuw geprobeerd.Berichten kunnen verloren gaan!Logging, statistieken — berichten die je wel kunt missen.
At least once
minimaal één keer		Het bericht komt zeker aan, maar kan meerdere keren afgeleverd worden als er iets misgaat (retry).Duplicaten! Zorg voor idempotency.Betalingen verwerken, orders aanmaken — mag niet verloren gaan.
Exactly once
precies één keer		Het bericht komt precies één keer aan. Geen verlies, geen duplicaten.Erg moeilijk en duur om te implementeren. Leidt tot veel overhead.Financiële transacties — alleen als at-least-once + idempotency niet volstaat.
⚠️ In de praktijk: "Exactly once" klinkt ideaal, maar is technisch heel complex en traag. Meestal kies je voor at-least-once + idempotency als compromis.
Idempotency — de sleutel bij at-least-once
Idempotent = hetzelfde bericht meerdere keren verwerken geeft exact hetzelfde resultaat als één keer verwerken.
Voorbeeld niet-idempotent (fout):
Bericht: "Schrijf €50 af van rekening X."
Als dit twee keer binnenkomt → twee keer €50 afgeschreven. Groot probleem!

Voorbeeld idempotent (goed):
Bericht: "Verwerk betaling met ID=abc123."
De ontvanger controleert: "Heb ik ID=abc123 al verwerkt?" Ja → negeer. Nee → verwerk.
Als dit twee keer binnenkomt → wordt maar één keer verwerkt. ✅
🔑 Hoe maak je iets idempotent?
• Gebruik een uniek bericht-ID (messageId / correlationId)
• Sla verwerkte ID's op in een database
• Check bij ontvangst: "Al verwerkt?" → skip als ja
📋 Verband met andere concepten
• At-least-once + idempotency ≈ exactly-once
• Acknowledgement (Ack/Nack) bepaalt of een broker opnieuw stuurt
• Poison messages kunnen ook duplicaten veroorzaken
Samenvatting: Delivery Guarantees
BegriffSimpel uitgelegd
At most onceBericht kan verloren gaan. Geen retry. Gebruik alleen voor niet-kritieke data.
At least onceBericht komt aan, maar kan dubbel zijn. Zorg voor idempotency aan de ontvangende kant.
Exactly onceBericht komt precies één keer aan. Technisch complex en traag. Zelden gebruikt.
IdempotencyZelfde bericht meerdere keren = zelfde resultaat. Realiseer met uniek message-ID + check.
Ack/NackAck = bericht verwerkt, verwijder uit queue. Nack = mislukt, probeer opnieuw (→ at-least-once).
13. Applicatielandschap — hoe werken systemen samen?
Wat is een applicatielandschap?
Een applicatielandschap is een diagram van meerdere systemen die met elkaar communiceren. Je ziet welke systemen er zijn, hoe ze zijn verbonden en welke data er stroomt.
Verschil met applicatiearchitectuur:
• Applicatiearchitectuur = hoe is één app van binnen opgebouwd (lagen, klassen)
• Applicatielandschap = hoe werken meerdere apps/systemen samen op systeemniveau
Voorbeeld: e-commerce applicatielandschap
Wat zie je in dit diagram?
• De webshop-frontend praat via REST met de backend API
• De backend slaat data op in een database en betaalt via een extern betaalsysteem
• Asynchroon: de backend plaatst berichten in een message queue
• Het legacy-systeem leest de queue asynchroon — geen directe koppeling!
Point-to-point vs hub-and-spoke (ESB)
Hoe koppel je meerdere systemen aan elkaar? Er zijn twee basispatronen.
🔗 Point-to-point
Elk systeem koppelt rechtstreeks aan elk ander systeem. Bij 5 systemen = 10 verbindingen.

Probleem: snel spaghetti! Als systeem A verandert, moeten B, C en D allemaal worden aangepast → tight coupling op systeemniveau.
🌐 Hub-and-spoke (ESB/Broker)
Alle systemen praten via een centrale hub (message broker of ESB). Systemen weten niet van elkaars bestaan.

Voordeel: loose coupling op systeemniveau. Nieuw systeem toevoegen = alleen koppelen aan de hub.
💡 Verband: Een message queue (RabbitMQ, Kafka) is een vorm van hub-and-spoke. Producers en consumers zijn los van elkaar gekoppeld via de broker.
Samenvatting: Applicatielandschap begrippen
BegriffSimpel uitgelegd
ApplicatielandschapDiagram van meerdere systemen die samenwerken. Toont welke systemen er zijn en hoe ze communiceren.
Point-to-point koppelingSystemen praten direct met elkaar. Simpel bij weinig systemen, spaghetti bij veel systemen.
Hub-and-spoke / brokerAlle systemen via een centrale hub. Loose coupling op systeemniveau.
ApplicatiearchitectuurHoe één app van binnen werkt (lagen, klassen).
SysteemarchitectuurHoe meerdere apps/systemen samenwerken. Zichtbaar in het applicatielandschap.
14. Referentiearchitectuur & de DAO-laag
Wat is een referentiearchitectuur?
Een referentiearchitectuur is een standaard-bouwplan op hoger/abstracter niveau: een vaste set lagen met vaste verantwoordelijkheden, waar je elke applicatie in onderbrengt. In dit vak werk je met 4 lagen.
Je hoeft het wiel niet opnieuw uit te vinden: élke app stop je in dezelfde lagen. Een nieuwe collega snapt je code meteen, want hij kent de indeling al.
De 4 lagen — wie doet wat?
De 3 lagen ken je al (sectie 3). De 4e laag (Data Storage) is nieuw en komt erbij zodra je data permanent wilt opslaan.
LaagWie zit erinVerantwoordelijkheid
PresentationGUI-componenten (met ActionListener)Alles van het scherm. Zo "dom" mogelijk.
Application LogicManagers (= service layer)Logica op app-niveau, checks ("mag dit?"), delegeert opslag.
DomainEntiteiten (Member, Book…)Weten alles van het domein: info + regels. Verder zo min mogelijk.
Data StorageDAO's, DatabaseConnection, DatabasePropertiesÁlle kennis over opslag: SQL, connectie, database.
💡 Regel: per persistente domeinklasse maak je één DAO. Naamgeving: <KlasseNaam>DAO → dus MemberDAO, BookDAO.
Verantwoordelijkheden — wie weet wat?
Manager weet zelf weinig, maar weet wél wáár hij info kan halen.
Op initiatief van de GUI haalt de Manager entiteiten op via de DAO en geeft die aan de presentation layer terug, die ze gebruikt om het scherm te tekenen.
Ontwerpvolgorde + iteratief werken
In welke volgorde bouw je de lagen? En hoe pak je het slim aan?
1
Begin met de Domain layer (het hart).
2
Dan de Application Logic layer (de managers).
3
Daarna óf de Presentation óf de Data Storage layer.
✅ Werk iteratief: bouw telkens een werkende verticale slice — een dun plakje door álle lagen heen met één klein stukje werkende functionaliteit. Dus niet eerst álle entiteiten, dan álle managers… maar: één feature helemaal af, dan de volgende.
ORM — opslag automatiseren
Een DAO-laag maken is bijna mechanisch werk (steeds dezelfde CRUD-acties). Dat kan dus geautomatiseerd worden.
ORM (Object-Relational Mapping) = een tool die automatisch je objecten ↔ database-tabellen koppelt, zodat je geen SQL met de hand hoeft te schrijven.
Voorbeelden: JPA, Hibernate, Entity Framework.
Een DAO heet bij een ORM een CRUD repository (CRUD = Create, Read, Update, Delete).
Meerdere databronnen → interface + DAO-variant
Wil je data uit een database én uit een file én uit een webservice halen? Twee opties:
❌ Eén DAO alles laten doen
De MemberDAO aanpassen zodat hij meerdere bronnen aankan.

Probleem: hoe weet hij welke bron actief is? Hij krijgt meerdere verantwoordelijkheden (schendt SRP).
✅ Per bron een DAO-variant
Eén DAO per bron, allemaal achter dezelfde interface.

Voorwaarde: de Manager mag er níéts van merken → "program to an interface". De Manager praat alleen met de interface.
15. RabbitMQ — de begrippen in detail
Wat is RabbitMQ?
RabbitMQ is een message broker: een soort postkantoor dat berichten van producers aanneemt en bij de juiste consumers aflevert. Het gebruikt het protocol AMQP.
AMQP (Advanced Message Queuing Protocol) is voor queues wat HTTP is voor webverkeer: de afspraak hóé producers en consumers met de broker praten.
Hoe een bericht door RabbitMQ reist
De RabbitMQ-begrippen op een rij
BegripSimpel uitgelegd
AMQPHet protocol voor queue-verkeer (vergelijkbaar met HTTP).
ExchangeHet "sorteercentrum". Nodig voor publish/subscribe — ontkoppelt producer en consumer in gedrag. Voor point-to-point gebruik je de default exchange.
Routing keyEen label dat de producer op het bericht plakt. De exchange beslist met dit label naar welke queue het gaat.
QueueDe wachtrij waar berichten staan tot een consumer ze ophaalt (FIFO).
ACK / NACKAck = verwerkt, weg uit de queue. Nack = mislukt, opnieuw proberen.
Dead-letter queueLukt een bericht na X pogingen nog niet, dan gaat het naar deze aparte queue. Daar kun je het inspecteren, opnieuw verwerken of weggooien — zo blokkeert het de gewone queue niet.
Point-to-point vs Publish/Subscribe
➡️ Point-to-point
Eén bericht → één queue → wordt door één consumer verwerkt.

Gebruikt de default exchange.

Voorbeeld: "GameEnd" → queue → UpdateLeaderboard.
📢 Publish/Subscribe
Eén bericht → exchange → meerdere queues → meerdere consumers krijgen elk hun eigen kopie.

Heeft een exchange nodig.

Voorbeeld: "GameEnd" → zowel UpdateLeaderboard áls NotifyViewers.
💡 Competing consumers: zet je méér consumers op dezelfde queue, dan "vechten" ze om de berichten — elk bericht gaat naar precies één van hen. Handig om de last te verdelen (schaalbaarheid).
Abstracties boven queues
Direct met queues werken betekent rommelen met bytes, verbindingen en kanalen. Er zijn libraries die dat makkelijker maken (een laag bovenop RabbitMQ e.d.).

16. Thin vs Fat events & Event-driven architecture
Thin vs Fat events — hoeveel data stop je in een event?
Een event kan weinig (thin) of veel (fat) data bevatten. Beide hebben voor- en nadelen.
🪶 Thin event (weinig data)
{ eventType:"GameWon",
 payload:{ gameId:"7791" } }

+ Klein schema → minder kans op schema drift (producer & consumer snappen hetzelfde).
+ Producer hoeft niet te bedenken wat consumers nodig hebben.
+ Minder stale data: consumer haalt zelf de verse data op.
− Te weinig data → consumer moet extra synchrone calls doen → herintroduceert temporal coupling, slechter schaalbaar.
🍔 Fat event (veel data)
{ eventType:"GameWon",
 payload:{ gameId, winnerId,
 username, finalScore, map…} }

+ Consumer is zelfstandig: geen extra calls nodig.
+ Betere fault tolerance: werkt ook als de producer even down is.
− Infrastructuur verwerkt meer data dan nodig.
− Data kan al verouderd (stale) zijn tegen de tijd dat hij verwerkt wordt.
Schema drift = producer en consumer raken het oneens over hoe een bericht eruitziet (bv. producer voegt een veld toe dat de consumer niet kent).
Wat betekent "event-driven"?
Een event-driven systeem reageert op wat er ís gebeurd, niet op wat er gevraagd wordt.
1
Systemen reageren op gebeurtenissen, niet op verzoeken.
2
Asynchrone communicatie → losgekoppeld in tijd én gedrag.
3
Events zijn first-class citizens: ze worden expliciet afgehandeld en gaan niet zomaar verloren.
4
Events representeren echte gebeurtenissen uit de buitenwereld (GameEnded, OrderPlaced).
Voor- en nadelen Event-Driven Architecture (EDA)
✅ Voordelen❌ Nadelen
Ontkoppeling in tijd (fault tolerance, recoverability) en gedrag (modularity, modifiability).

Beter schaalbaar (performance, availability, capacity).

Event = first-class citizen → gaat niet verloren. 		In zijn geheel lastiger te debuggen.

Vraagt andere ontwerpkeuzes (wanneer is welke info beschikbaar?).

Vereist backwards compatibility → meer complexiteit.

Werkt het best met speciale infra (queues) → meer complexiteit. Volgorde lastiger te coördineren.
Verwante "event"-begrippen (verschil kennen!)
De term "event" komt op meerdere plekken terug. Weet hóé ze verschillen van event-driven architecture.
BegripSimpel uitgelegd
Eventual ConsistencyData is niet meteen overal gelijk, maar wordt het uiteindelijk. Gevolg van async werken: even geduld, dan klopt het overal.
Event SourcingJe slaat niet de huidige toestand op, maar álle gebeurtenissen die ertoe leidden. De toestand bouw je op door de events opnieuw af te spelen.
